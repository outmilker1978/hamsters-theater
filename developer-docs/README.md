# TV Hamsters — Документация разработчика

**Версия продукта:** 1.8.2  
**Дата:** июнь 2026

---

## 1. О продукте

TV Hamsters — десктопное приложение для совместного просмотра видео с видеочатом,
демонстрацией экрана, передачей системного звука, чатом и emoji-реакциями.

Работает через локальную сеть (LAN) или через Интернет (облачный сервер на Render.com).
i18n: RU/EN/ES. P2P mesh до 5 участников.

## 3. Архитектура

### Локальный режим
На ПК хоста запускается встроенный Socket.IO сервер (порт 3000+, auto-fallback).
Гость подключается напрямую по LAN IP:порт. UPnP пытается открыть порт на роутере.

### Облачный режим
Оба ПК подключаются к серверу на Render.com (tv-hamsters-bot). Сервер только пересылает
сигналы (offer/answer/ICE). Видео/аудио — напрямую P2P через WebRTC + STUN (Google Public).
TURN не используется.

### ICE restart
При 4 последовательных плохих замерах качества (20 с) вызывается `restartPeerConnection(pid)`,
который пересоздаёт PC через `createOfferToPeer()`. Не трогает screenPC. Кулдаун 60 с.

```
ПК1 ←—— WebRTC (P2P) ——→ ПК2
  │                         │
  └── Socket.IO ——→ Render.com ——→┘
```

## 5. Облачный сервер + Telegram бот (Render.com)

Все функции объединены в `bot/index.js`: Socket.IO сигналинг + Telegram бот.

**Адрес:** https://tv-hamsters-bot.onrender.com
**Репозиторий:** https://github.com/outmilker1978/hamsters-theater

### Обрабатываемые события

| Событие | Назначение |
|---------|-----------|
| `create-room` / `room-created` | Создание комнаты (4-значный PIN) |
| `join-room` / `joined` / `peer-joined` / `user-joined` | Подключение к комнате |
| `offer` / `answer` | Обмен SDP (WebRTC) |
| `ice-candidate` | ICE кандидаты |
| `signal` | Прикладные сигналы (screen, PTT) |
| `chat-message` | Чат |
| `reaction` | Emoji-реакции |
| `disconnect` | Очистка при отключении |

### Telegram бот
- **Команды:** `/start`, `/create`, `/room`, `/help`, `/download`
- **Автосоздание комнаты** при ключевых словах в групповых чатах
- **Ссылка:** t.me/tv_hamsters_bot
- **Токен:** в GitHub Secret `TELEGRAM_BOT_TOKEN`

### Как управлять

1. Зайти на https://dashboard.render.com (войти через GitHub)
2. Выбрать Web Service `tv-hamsters-bot`
3. Доступно:
   - **Logs** — логи сервера
   - **Manual Deploy → Deploy** — перезапуск
   - **Auto-Deploy** — автообновление из GitHub

### Важно

Render бесплатный — сервер засыпает после 15 мин бездействия.
При подключении просыпается за 5-15 сек. Для постоянной работы — $7/мес.

### Как обновить URL

URL задаётся в:
- `bot/index.js` — токен Telegram бота и порт
- `renderer/app.js` — через IPC из main.js
- `docs/mobile/app.js` — константа CLOUD для мобильной версии

## 6. Сборка .exe

**Требования:** Node.js v24, npm 11, зависимости установлены (`npm install`)

```bash
# Полная сборка (создаёт dist/win-unpacked)
npm run build

# После сборки — портативный .exe из свежего dist/win-unpacked
npx electron-builder --prepackaged dist/win-unpacked --win portable
```

**⚠️ --prepackaged НЕ обновляет код** — всегда сначала полная сборка.

**Важно:** `npm run build` падает на winCodeSign (симлинки) — НЕ критично. packaging проходит успешно.

**Проверка:** `(Get-Item dist\TV.Hamsters.*.exe).LastWriteTime` должен быть после изменения кода.

Результат: `dist\TV Hamsters X.X.X.exe` (autoname — новый номер для обхода Windows Defender)

## 7. Разработка

### Запуск для отладки
```bash
npm start          # запуск Electron
F12                # DevTools
```

### Версионирование
Перед сборкой обновить версию в:
1. `package.json` — `"version": "X.X.X"`
2. `renderer/app.js` — `const ver = 'X.X.X'`
3. `renderer/index.html` — `id="versionDisplay"` и `class="modal-version"`
4. `renderer/i18n.js` — release notes ключи (RU/EN/ES)

### Архитектура multi-user
- **Mesh-топология:** каждый клиент создаёт отдельный RTCPeerConnection для каждого участника
- `peers = {}` — карта peerId → { pc, screenPC, remoteStream, cameraCandidates, screenCandidates }
- Новый участник рассылает `user-joined` всем в комнате, каждый существующий создаёт offer
- Screen sharing: один screenPC на каждого пира (транслируется всем при старте)
- **Чат:** три IPC-канала (main-chat-send, panel-chat-send, forward-chat). Сервер: socket.to(roomId), без эха отправителю
- **Реакции:** emoji на весь экран (reactions.js), анимированные взлетающие эмодзи
- **Скример:** команда /скример в чате → img.onload + таймер → скример-страница со звуком
- **Индикатор качества:** `peerQualities[pid]`, обновляется каждые 5 с. Цветная точка 10px с border+glow
- **ICE restart:** `poorQualityCount[pid]` >= 4 → `restartPeerConnection(pid)` → `createOfferToPeer(pid)`. Кулдаун 60 с
- **PTT:** правый клик по микрофону → режим PTT. Space (глобально), кнопка PTT (mousedown/mouseup)
- Сервер: лимит комнаты 5 участников. Шлёт и `peer-joined`, и `user-joined` для совместимости

### Обратная совместимость
⚠️ **Обязательное правило:** каждая новая версия должна сохранять совместимость со старыми клиентами. Если это невозможно (например, ломается протокол сигнализации), сервер должен поддерживать оба формата событий, и клиент должен слушать оба варианта. Если совместимость принципиально невозможно сохранить — **сообщи пользователю перед сборкой**.

### Важные моменты
- Изменение `server/cloud.js` → push на GitHub → Render перезапустит сам
- Изменение `main.js` или `renderer/*` → пересобрать .exe
- Формат: семантическое версионирование (major.minor.patch)

## 9. Совместимость браузеров (Mobile PWA)

WebRTC — базовая технология видеосвязи в браузере. Она работает на движке Chromium,
который используется в Google Chrome, Яндекс.Браузере, Samsung Internet, Opera и других.

**Стабильно работает:** Google Chrome и PWA на его основе.
**Возможны проблемы:** Яндекс.Браузер (встроенный VPN/Турбо, блокировка TURN, нестандартные разрешения).
**Не поддерживается:** Internet Explorer, устаревшие браузеры без WebRTC.

## 10. Известные проблемы

- **`let` vs `var`:** каждый `<script>` в Electron — отдельный scope. `let currentLang` не виден из `app.js`. Использовать `var` для кросс-скриптовых переменных
- **Screen share freeze:** когда источник свёрнут, Chromium может не обновлять кадры (ограничение браузера)
- **Yandex Browser:** встроенный VPN/Turbo ломает P2P (STUN не видит реальный IP)
- **WiFi роутеры:** mesh-топология нагружает NAT-таблицу. Переключение одного участника на 4G/5G снижает нагрузку
- **winCodeSign:** не извлекается на Windows (симлинки) — НЕ влияет на portable сборку
- **Render бесплатный:** засыпает после 15 мин бездействия
- **Модалки i18n:** статичны в HTML (display: none), перевод только при вызове `setLang(currentLang)` при открытии
- **Windows Defender:** блокирует старый .exe при перезаписи. Новый номер версии в имени файла решает проблему
